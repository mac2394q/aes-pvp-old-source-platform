System Development and Implementation Projects
Applicability of Standards based on type of Project

Table A below should be used to determine the type of the project i.e. Small, Medium or Large. Any project which has any two attributes in a higher category should be classified in that higher category e.g. if a project using proven technology in a stable environment, has only one IT resource assigned and the number of impact users is just 5 but if there is a capital outlay of $105,000 and the project is going to take 4 months to complete, the project must be classified as a Medium size project, not Small.

1. Project Management

1.1. All medium and large projects must be presented at a Project Initiation Meeting prior to being communicated to the business. This is a recommended step for small projects. The meeting objective is to communicate staffing assignments, impact to other initiatives, set date for estimate review and obtain signoff to proceed.

1.2. The project should be conducted under the rules and guidelines of good project
management practice. The project manager should choose an approach at the
preliminary design stage of the project, once the business requirements have been
developed and solidified. The types of good project management practices that may be considered include:

Capability and Maturity Model for Software (CMM-SW)
ISO 9001 Quality Level for Software Process
PMI Project Management Body of Knowledge (PMBOK)
The Rational Unified Process (RUP) Adaptable Process Framework and the Unified Modeling Language (UML)
Projects in Controlled Environments (Prince2) UK Standard for Managing Projects
Control Objectives for Information Technology (CObIT) Management Guidelines


1.2. The project should follow a templated approach to project management as ascribed in one of the above methodologies and all participants should be familiar with the process being used. If steps in the process are to be bypassed, an addressable and step in the project plan should denote the reasons why this is being done

1.3. Projects that are rated as medium or large per the standard criteria are required to have a steering committee or implementation board. It is highly recommended that smaller projects also maintain a steering committee structure but it is not a requirement. The primary purpose of the steering committee is to oversee the planning and ensure success of the associated project. The steering committee should be formed during the requirements gathering phase and provide sign-off on the project requirements. It should end when the project has been successfully implemented and all significant outstanding issues associated with the project have been closed.

Role of the Steering Committee:

        • Meet on a regular basis until the project is brought to closure and key post-production issues are identified and resolved.
        • Secure the budget for the project
          • Review the total cost of ownership
          • Set criteria to be used when determining if change requests require steering committee approval (dollar amount, time impact, etc.)
          • Monitors scope, reviews and approves scope, schedule and budget modifications in a timely manner
          • Stop projects that will not bring the required value to the company
        • Secure the resources for the project
          • Ensure that resources are used correctly
          • Ensure qualified project management
          • Provide mentoring and support
          • Celebrate success and rewards
        • Assist project manager to ensure success of the project
          • Provide expertise as needed
          • Ensure that issues identified are properly addressed and closed
          • Facilitate communication with upper levels of management
          • Liaise with the user community
        • Support the project governance process
          • Assess risk and mitigation
          • Make decisions to keep the project tracking properly
          • Ensure all gating and review processes are followed
          • Provide support and ensure adherence to all audit and security requirements



1.4. All implementation team roles, responsibilities should be clearly defined and documented prior to the onset of the implementation phase

1.5. The objectives, scope and business requirements of the project should be clearly defined and documented. If the implementation involves heavy vendor / consultant involvement, the statement of work should be signed off with the vendor / consultant prior to commencement of implementation / development.

1.6. The future Application Owner, IT Application Manager and Security Administrator (Refer to Section 4.1, Application Security Standards) should be identified and included in the project team.

1.7. A detailed project plan should be developed (use of tools like MS project are preferred) for the development / implementation project, which outlines the different tasks, milestone and resource allocation.

1.8. Critical milestones need to be identified and the project steering committee has to monitor those milestones. The steering committee needs to review and approve the completion of different phases of the project. Minutes / other documentation of such meetings need to be maintained.

1.9. All project related documentation should be stored in a central repository accessible to all project members and sponsors. Use of collaboration tools like Quickplace are encouraged for this purpose

1.10. A log of all issues, problems and risks should be maintained and tracked. The status of the issues and risks should be clearly marked. Issues and risks must be prioritized and escalated to the Steering Committee and sponsors as required.

1.11 Any project utilizing any new technologies (hand-held devices, RFID, wireless) need to be communicated to ITS security and Internal Audit during the feasibility stage to ensure appropriate input and determine if involvement in the design / implementation stages is required.

1.12 Testing procedures employing elements such as individual test cases, parallel processing, volume testing and total system testing should be employed to ensure the proper operation of new and changed programmed procedures and the proper effect on unchanged programmed procedures. Testing should include internal and external interfaces.

1.13 Development, Testing, Training and Quality Assurance environments need to be created as appropriate and these need to be segregated from the production environment. Appropriate environments must continue to be maintained after implementation for development, training and testing purposes.

1.14 End users must be trained in the new systems and training should be available for new hires or when users require retraining. Users should have access to training documentation. The end users should define and develop end-user testing scenarios and sign-off on final testing.

1.15 Functional, technical and user documentation should be developed and maintained during the project implementation life-cycle. This documentation should be turned over to the IT Applications Manager at the end of the project. The IT Applications Manager is responsible for disseminating this documentation appropriately.

1.16 If the project involves data conversion from an existing system, the project plan should include phases for data mapping (including updating meta-data) and gap analysis. Further, data cleansing and validation should be performed to ensure data accuracy, integrity and relevance. Necessary validations (e.g. hash totals) need to be made to match the converted data with the data in the earlier system.

1.17 The end users of the system should sign off and accept the new system prior to the system being put in production.


2. Systems Development- Methodology


2.1 A systems development project methodology should be defined and utilized for all projects with these life-cycle phases:
            • Project initiation
            • Functional design analysis and planning
            • System design specification
            • Software development
            • Installation/implementation
            • Operational/maintenance
            • Revision or Replacement

And these essential components:

            • Requirements analysis
            • Preliminary design
            • Detailed design
            • Coding
            • Unit testing
            • Integration
            • System test (including parallel test, where feasible)
            • Training


2.2. E-commerce applications accessible via the Internet should undergo a thorough review by the ITS Security group prior to the implementation phase and again prior to the movement from test to production. A vulnerability scan against the application and its resident components should also be done prior to the application being entered into production

2.3 Prototyping should be used, where possible, in the development of systems. A specific advantage of a prototype is that it can serve as a means of communication between customer and developer. A prototype should be developed in an environment where an acceptable, functioning and easily modified system can be created.

3. Systems Implementation - Methodology


3.1 For projects whose total cost to implement exceeds $1 million USD, software customization should be done only when the base package cannot support the needs of the business and is approved by the project steering committee or program management team and the Chief Information Officer (CIO)

3.2 All systems implementations over $1MM should undergo at least one design review with the system's stakeholders / Implementation Board prior to implementation


4. System Development / Implementation Security and Control Configuration

4.1. Application level passwords should not be coded into the application itself nor hard-coded during any development phase of a project. Passwords should be stored in encrypted form in a separate, secured file. If a password is used for app-to-app or app-to-database authentication, the password should be encrypted and stored in a file or repository, not in the application code or script. This file should preferably be readable only by the application run-time ID and a mechanism for changing this password periodically should exist.

4.2. The application must meet or exceed Praxair standards for passwords (refer application standards). Passwords must be stored in encrypted format when stored and transmitted.

4.3. The security design should be such that it allows for segregation of duties among users of the application. The design must allow for the segregation of inquiry, update and delete functions.

4.4. The application should record user-id/ timestamps for transactions or updates to master files which enables identification of the user who entered the transaction or last modified transaction data. Where data is being received through an interface from another system, such transactions need to be clearly identifiable to the sending system and also contain timestamps/ audit trails from the sending system. The applicable must be capable of providing logs of such interface transactions. The application owner should perform a risk analysis to determine critical or sensitive areas where audit trails should be utilized

4.5. The application should have reporting capabilities where access can be easily understood and reviewed

4.6. The system should have inquiry and reporting capabilities that provide audit trail and timestamp links to transactions.

4.7. If feasible, the application should be made to run in user mode and without any privileges.

4.8. Network/Server accounts for application should not be set to allow full control, however use of service accounts to perform specific functions outside of application control may be utilized.
4.9 Where applicable, input screens should include default values for input data and have reasonableness checks for data (e.g. date fields, limits for values), checks for completeness of input and for duplicate data.


5. Systems Development / Implementation - Infrastructure Elements


5.1. As part of the feasibility study, the following need to be considered:
5.1.1. Capacity and performance requirements
5.1.2. Availability requirements
5.1.3. Data retention requirements
5.1.4. Database growth and archiving

5.2. The Business Impact Analysis is needed to determine the implications of the loss of access or functionality of an application due to an unforseen event such as a natural disaster, accident or act of terrorism. Please see Section 4.5, Backups and Disaster Recovery, for more details.

5.3. ITS Infrastructure standards must be followed for building the servers for the application and all required patches must be applied to the servers during development / implementation.


5.4. The following should be evaluated by the Sponsors and decisions documented regarding backups

5.4.1. Frequency of backups and type of backups (full, incremental)
5.4.2. Length of backup retention
5.4.3. Encryption requirements for backups

5.5. Backups of the servers need to be taken periodically during the implementation and if the project is planned to run parallel and then cut-over, the same standards as production backups should be applied.

5.6. At least one full testing of restoration from backups should be conducted as part of system testing

5.7. Testing of the client application must include testing in all the different desktop environments of the intended users, with security patches applied.

6. Systems Development / Implementation - Resource Management


6.1 A risk analysis should be conducted in the functional design specification to determine that the right resources and personnel are engaged or are available for this project/system

6.2 If data in the application system will be of a confidential nature, the IT application manager should ensure that the resources involved have been approved for access by the application owner
6.3 Participants in a software development project should maintain separation of duties so developers do not have access to code used in production, except where this is expressly approved by the application owner. However, development personnel should not be granted administrative rights to the application or the database.

6.4 Developers should maintain separate logins for "user" access and "development" access. Under ideal conditions, the code generated should be
tested, submitted by the developer to the library and then sent to the production environment.

6.5 Developers should be trained ahead of the detailed design phase of a project so that they are well versed in the construction (design/implementation) of a deliverable system

6.6 If vendor support is required, this should be restricted to only the required resources. Unrestricted access to the production environment must not be permitted. A project team member must monitor vendor access.


7. System Development/ Implementation - Support

7.1. Any developed application should have, at minimum, a primary and back-up administrator capable of successfully executing all of the support functions for that system. A training curriculum for these administrators should be maintained to keep them current with changes in the application and the technology being used.
8. System Development / Implementation - Operations Controls

8.1. Application systems must include control procedures to ensure that:

8.1.1. Only the business transactions that have been properly input and verified are processed

8.1.2. Business transactions passed from one application system and processed by another system have been properly transmitted by the source system and received and processed by the receiving system. The application must provide logs of the interface jobs.

8.1.3. Concurrent file or database updates are properly managed

8.2. Application developers must prepare specific procedures for restarting processing in the event of a temporary hardware or system failure where applicable. These recovery procedures must be provided to the responsible data centers operations personnel.
9. Change Control-Applications

9.1. Requests for maintenance changes should include the business reason and provide verification of written notice to the application owner (can be done via e-mail) with their approval and the documentation (electronic form can be used) should be maintained by the IT Applications Manager. At minimum, for a change control process:


9.1.1. Formal written (electronic or paper) request of change must be made
9.1.2. The request should be analyzed by the submitter for implementation strategy, cost, security implications etc.
9.1.3. A record of that change request should be maintained with a unique identifier
9.1.4. The request should be submitted to the change control committee
9.1.5. The committee will approve or disapprove the request, disapproval will be accompanied by a reason and the submitter can cure and re-submit

9.2. The change implementer should develop the change following all other change control standards and system development methodology

9.3. Changes should be controlled in such a way that it can be uniquely accounted for and distinguished from other application changes (e.g. numbering scheme or time/date stamp)

9.4. Emergency change controls made to applications that bypass normal procedures need adequate after-the-fact notification procedures, including written notification to the application owner

9.5. All changes should be tested, where possible, in test conditions and in the test environment that will ensure the system operates as intended and errors in processing are not created by the change

9.6. All documentation must be updated to reflect the scope and impact of system changes

9.7. Production software and data files must be segregated from versions used in testing

9.8. If distributed systems are used on more than one computer, system-wide version controls must be developed to ensure proper versions of programs and data files are used throughout the system

9.9. Prior to implementation, the relevant factors concerning application changes should be reviewed by IT management for their concurrence or reviewed by a change control committee, where such a committee can be formed. Application changes should not, under normal conditions, be approved by the manager of the submitter of the change since this represents a conflict of interest

9.10. If changes to source code are warranted as part of the change, these changes must be made to test code which is then migrated into production, never directly into the production code

9.11. The transfer of software changes from test to production status must be controlled and documented

9.12. Changes must ensure that the security policy can still be implemented and that the security mechanisms have not been negatively affected

9.13. An audit trail of changes made to production applications must be retained for a reasonable period to provide the ability to adequately verify the appropriateness of the change and recreate prior versions

9.14. After major system changes

9.14.1. Technical and User documentation needs to be updated, and

9.14.2. Users must be trained in the changes

9.14.3. Both the documentation and training must be available to the users

10. Go-Live Checklist

        • The Go-Live Preparedness Checklist should be completed for all medium and large projects. The project manager is responsible for ensuring this document is completed, presented at a project steering committee meeting and approved by the business owner prior to go live. The completed checklist along with the supporting documentation must be attached to the Global Applications Inventory. Any additional documentation related to the checklist such as email sign offs, should also be attached to serve as evidence for SOX 404 testing if applicable and for internal audits. While certain items in the checklist are marked as 'USA only' as they relate to specific processes in the US, these items are also highly recommended for similar processes globally.

11. Administrative Function Controls
          • All administrators (e.g. database, LAN, application) should be required to sign a Code of Conduct document that will be retained by ITS Security outlining the consequences of abusing the elevated rights and privilege level afforded these employees
          • At minimum, a single back-up administrator should be trained and equipped to fill in for the primary administrator on each system, database or application requiring support or implementation
          • Background checks should be performed on each person requiring administrative access unless the local laws of that country prohibit this type of verification process. This should be done either at the time of employment or upon transfer into an administrative function role.
12. Discontinuance of Maintenance

Approval must be obtained from the CIO prior to the discontinuance of maintenance, on any application categorized as medium or large based on the IT Control standards. This includes applications for which the vendor is discontinuing support. The approval request must include the risk factors associated with dropping maintenance, proposed alternative support resources, potential cost savings and a road map outlining the potential life span of the application at Praxair including expected growth/decline of use over time. All approved requests should then be communicated to procurement by the requester.
