IT Standards Infrastructure

4.2 Baseline Infrastructure Standards

Policy Statement: ITS is the custodian of all computing, network and telecommunications equipment and related software and is responsible for imaging and security controls on PC�s (e.g. anti-virus, personal firewalls and security patches). End-users are the custodians of PC�s, and are responsible for the due care of each machine and other software not provided in the standard image. Unless otherwise specified in the standard, ITS is responsible for compliance with the standards below.

A. Network

1. Persistent External Connections

1.1. Each external connection should have an associated business sponsor that will be recorded by ITS Network Services or ITS Security

1.2. ITS network teams are to prepare and keep updated all documentation to enable review and evaluation for continued use

1.3. Any connections to the Internet must be protected by an ITS Security approved firewall (refer to document, �Currently Approved Technologies for Security�), which is configured and administered based on ITS Security guidelines (link to Firewall Standards)

1.4. Internet based connections such as VPN�s, MPLS, IP Frame Relay, XML EDI etc. should be configured in accordance with ITS security guidelines

1.5. Non-Internet based external connections (e.g. frame, leased lines, private ip) should, at a minimum, have access controls implemented per Network and ITS Security guidelines, and should be reviewed and approved by ITS security prior to implementation

1.6. All network connections are to be reviewed on an annual basis by IT security in conjunction with the IT network team and the appropriate IT Applications Manager

1.7. Servers to be used for e-commerce or web-hosting should reside in a De-Militarized Zone (DMZ) or Screened Sub-Net (SSN) with the external user having no direct access to the internal Praxair network. E-commerce servers should be configured in accordance with the E-commerce security policy as set forth by IT Security. Scanning of servers residing in DMZ's or SSN's should be conducted on a twice annual basis, at minimum and should be done using a tool approved by Global ITS Security.

1.8. Any connections to non-Praxair owned third party/e-commerce sites that use Internet technology for transport should be reviewed and approved by IT Security prior to implementation, and data transport should be ordered and installed under the auspices of the IT Network Services group

1.9. All confidential information transmitted over external (non-Praxair managed) networks must be done in encrypted form using tools specified in the Praxair Information Security Policy

1.10 Use the following IKE and IPSEC values for all VPN connections between company standard firewalls (see Supporting Standards for more information):
          • IKE (Phase1) 3DES Encryption, Data Integrity MD5 or SHA1. DH Group 2 (ie 1024 bit), Renegotiate SA every 1440 minutes. On the firewall configuration select both MD5 and SHA1 and let the firewalls negotiate.
          • IPSec (Phase 2) Renegotiate SA every 3600 seconds. For the VPN tunnel use 3DES encryption, MD5 data integrity, no compression, no PFS
          • Pre-Shared Secrets are never to be divulged in public and will be sent, securely, by Global ITS Security, only when required
          • Supported symmetric encryption algorithms are: AES, 3DES and DES
          • AES or 3DES is strongly recommended for all connections
          • Peer IP Addresses and Encryption Domains are determined at time of setup on the VPN by Global ITS Security and the administrator on the other end of the VPN connection

2. Transient External connections

2.1 Modem External

    • Each dial-up connection request should be done in writing via this form (attach form), approved by the employee�s manager and referred to the appropriate IT Application Manager so that it can be tracked on an annual basis by the IT Applications Manager, who should confer with the employee�s manager to ensure the on-going business need

    • Connections should only be made for the designated purpose in the business request and for only the duration needed to complete the transaction being run. The connection should be automatically (by configuration) or manually terminated (by user) once the transmittal/reception is ended

    • Dial-up modem use is only permitted if cleared and approved by local IT Management in advance

    • Use of modem firewalls is permitted and IT Security can specify a device for those who would like to use such a protective mechanism. The line disconnect requirement can be avoided by use of such a firewall

    • Dial-up access to the Internet (e.g. ISP accounts) from any Praxair system or computer is not permitted under any circumstances even if physically disconnected from the company network

    • The phone lines plugged into modems must terminate on a PBX or Key System, should be non-published in any corporate directory and in a numeric range different from that of normal desktop extension numbers. Dial-up phone line numbers must not be listed in any internal or external phone directories.

2.2. Modem Use to the Praxair Network

    • Modem use to the Praxair network is permitted, but only through approved mechanisms (Currently Approved Technologies for Security) as determined by IT Security and from Praxair provided equipment

    • From an external source to an internal server/PC, modem access needs to be approved prior to it being implemented. The server configuration should have this access turned off by default, it should be turned on only when access is required (therefore it is not available when access is not required) and a log should be generated for capturing the time and duration of the connection


3. Remote Access

3.1. Remote access is not granted by default with receiving a Praxair computer and a sign-on to the Praxair network and the assigned computer has a modem in it. The request for remote access must be approved by the manager of the requesting employee. This applies to contract and third-party personnel as well.

3.2. If Praxair security detection systems identify access attempts that appear to be of an unusual nature, IT Security reserves the right to terminate that connection and disable that account from use. In those cases, IT Security will attempt to contact the user whose account this access represents to ascertain whether the access is from the legitimate account owner. If the user contacts IT via the Help Desk or some other method, they could be asked for other information and only after the verification process has been validated will IT Security allow for the reinstatement of the account.

3.3. The mechanism or product used to grant such access is determined by IT security. The configuration parameters are set by guidelines determined by IT Security and technology specifications for various forms of remote access (e.g. VRAS, NT RAS, Generic Dial-Up) have to be reviewed by IT security prior to deployment

3.4. All remote access should provide for reliable and complete authentication and be logged, and the logs should be reviewed on a periodic basis by local IT Management

3.5. Split tunneling, the ability to access the Internet bypassing Praxair security controls while connected to the Praxair network via VPN, is not permissible
3.6. Employees must not attempt to circumvent any device (software or hardware) used to facilitate their secure remote access to Praxair systems

3.7. Except for IT Security approved devices, all remote access must be conducted using a Praxair provided device or methodology (ex. go.praxair.com)

3.8. When users are not actively using remote access capabilities on Praxair�s network infrastructure, the remote system should be logged off the network

3.9. To enable users to have their passwords reset when needed, the Help Desk associate should call back to the requester�s voice-mail and leave the password as a voice message. For locations lacking voice-mail, the Help Desk should call back the requester at the number they are at, and ask for the requester�s employee number or additional information that only the employee could provide (e.g. manager�s name or department name). An e-mail confirming the request (not containing the password) should be sent to that employee in these cases

3.10 The use of RAS as an on-going method of access must be done with the approval of Global IT Security. This should include the number of users, what method are they accessing by (cellular, PSTN dial or other) and how many RAS modems are in place.

4. Wireless Networking

4.1. Use of Wireless communication mechanisms must be approved for use by local IT Support prior to implementation. Local IT Support must verify such use with Global IT Security prior to granting permission. Employees must not use such technologies to access any company networks or with any Praxair provided equipment without first getting permission as defined

4.2. Personal or home based wireless networks are not recommended for use when connecting to any Praxair network, however, for situations where use of a home-based wireless network cannot be avoided, IT Security will provide, upon request, a guideline on how to operate your wireless network in a more secure manner. No support of home wireless networks is implied or should be assumed by any employee using one.

4.3 Non-employees of Praxair are not permitted to utilize any installed wireless facilities unless doing so from a Praxair managed computer or device.

4.4. Proper configuration and use of wireless local area network network functionality may be found in the IT Supporting Standard, Section 6, Sub-Section
6.4, Wireless LAN Policy.


5. Configuration and Procurement of Network Devices

5.1. Network devices (firewalls, hubs, routers, switches, modems) should be configured based on Praxair standards / procedures as defined by Global IT Security

5.2. Procurement of these devices should be done via the IT Network Services group and Global Procurement services. No deviation in product type is permitted without the consent of IT Network Services or Global Information Security

5.3. The configuration of network equipment (except firewalls), systems and software are the responsibility of local infrastructure management and should not be undertaken by anyone except those designated and known by IT Network Services and Security management
6. Access to Network Devices

6.1. Access to network devices should be controlled and restricted to the personnel administering the devices.

6.2. Access to routers and switches should be granted in accordance with the appropriate Security Guidelines for those devices

6.3. The list of personnel with administrative access to network devices should be kept and maintained by the IT Security group, changes will be provided to IT security by all infrastructure teams as they occur

6.4. Systems management tools such as HP Openview should be utilized where feasible to act as a central administrative point as well as an event correlation point for SNMP-like data. Access to these tools should be limited to IT Network or Operations personnel.

6.5. Access to routers and switches should be done via Secure Shell or Secure Sockets Layer and not Telnet. Telnet login information is transmitted in the clear and can be compromised

7. Network Protocol

7.1. TCP / IP is the standard LAN network protocol suite for Praxair data networks. Any exceptions need to be approved by Global IT security

7.2. Wide area network protocols may include: frame relay, Private IP, T-1/T-3, Public IP, Multi-Protocol Label Switching, VPN. Others beyond those listed must be approved by Global ITS Security

7.3. VPN configuration parameters will be provided by Global IT Security for a minimum configuration for both client based and site-to-site VPNs

7.4. Simple Network Management Protocol (SNMP) is to be used for all collection and correlation of events from network devices. SNMP community strings should be modified from their default settings and should be set to read-only so no modifications can be enabled by the protocol


8. Third Party Access

8.1. All third party access should be approved in writing by the application owner, identifiable by distinct naming conventions and restricted to the resources that they require using access controls (e.g. firewall, router) and server level file and rights permissions

8.2. Remote access by third parties may be granted, however, the third party must use the company preferred remote access method and cannot bypass this method to gain access to a server or system

8.3. For network equipment such as a router, a Praxair IT Networking employee should supervise and observe the work being done by the third-party contractor. Passwords, if given to the contractor, should either be set up as temporary passwords removed after that use, or should be changed immediately upon exit of the contractor from that system

8.4. For applications supported by third parties, administrator access for servers can be provided on a temporary, controlled basis only. Such access is to be requested by the Application Manager with timing agreed upon with IT infrastructure. The access must be disabled in a reasonable timeframe upon the third party�s disconnection. A server audit log must be generated and reviewed by IT Application Manager or their designate once the activity has been completed and the access disabled. Application changes should also be logged and reviewed as specified in the Praxair application standards.

8.5. All third party accounts should be set to expire every 180 days and reactivation would require management approval

8.6. The Praxair application manager hosting the third party is responsible for alerting IT (via the Help Desk) if the third party access can be revoked prior to the completion of the current 90 day window

8.7. Non-disclosure agreements need to be signed prior to granting access. Original copies should be forwarded to and stored by the IT Security group or the regional IT group responsible for security.

8.8. All third party personnel will be presented with and required to sign (or have an authorized representative of their organization sign it) a Contractor Acceptable Use Policy (attached here) that outlines the terms for access to Praxair networks by a third party entity. This may include security software such as anti-virus, personal firewall, and content scanning

8.9 Third Parties should never be granted administrative access to any system or network, except in cases where the function that the third party was enlisted for could not be achieved without said access. Even then, special steps should be taken to secure that access such as monthly reviews of all work done by the third party, change control logs of any administrative changes done to production systems , status reports crafted by the contractor, oversight of an employee administrator, etc. They should also not use their administrative credential when it is not needed and should log in only as user when user level functions are being performed

8.10 The use of personally owned computers by contractors is permissible in certain circumstances, but not recommended. Any non-Praxair computer used to connect to Praxair networks must meet the requirements outlined in the Contractor Acceptable Use Policy, which will read as follows:
      • Praxair provides access to the Microsoft Virtual Private Network device for access by those contractors who are required to remotely access a Praxair network.
      • Praxair requires that users who utilize this VPN solution do so from company owned and maintained computer systems, including laptop or desktop systems, except for rare circumstances whereby access to an Praxair computer is infeasible or impractical.. If anyone has questions about this policy, they may contact the Information Security Office at 203-837-2221
      • The computer used to access the Praxair network must have an up to date anti-virus program loaded on it, preferably TrendMicro PC-Cillin or OfficeScan. It must also be accompanied by a personal firewall such as, but not limited to, Trend Micro, CheckPoint Zone Alarm, Symantec Client Firewall or a hardware firewall such as, but not limited to, SonicWall or LinkSys. No efforts to defeat or disable these packages while connected to the Praxair network is permitted, even if said protection prevents connection, without the consent of local or regional IT management.

9. Monitoring

9.1. Network connection devices (firewalls, routers) will have auditing / logging enabled and the logs should be regularly (at minimum, once every 10 business days) monitored by the IT network teams.


10. Network Account Management

10.1. All accounts on the network must be authorized and uniquely identifiable. All exceptions and use of �generic� accounts should be approved by IT security prior to use.

10.2. All user accounts (user-ids) should conform to an IT specified global naming convention (id nomenclature) to facilitate the use of good account management practices. This nomenclature is to be determined.

10.3. Standards regarding passwords for all Microsoft based systems are as follows:

    • Minimum length of 6 characters
    • Users must be forced to change passwords after logging on the first time, including contract personnel
    • Accounts set to lock out after 10 consecutive unsuccessful attempts
    • Accounts will reset after 60 minutes
    • Passwords should be set to expire every 90 days
    • Passwords should not be the same as the username and should be difficult to guess
    • Passwords should not be shared
    • Passwords should not be reused
    • Passwords should not be written down and should be changed immediately on suspicion of compromise
    • All vendor supplied default passwords should be changed immediately upon use
    • IT may deploy user self service tools for resetting passwords by phone or kiosk service, but only after review and approval of IT Security

10.4. Procedures should be in place to delete accounts when users are terminated. Only when indicated by Human Resources or by the Law Department should this procedure not be executed.

10.5. Procedures should be in place for an annual review by Infrastructure of all users accounts to ensure that accounts no longer required are identified and deleted and accounts not active for 180 days are disabled and potentially deleted. The list compared against should come from the Human Resources Department.

10.6. User IDs set up for specific utility purposes (ex. Testing, training, presentation, troubleshooting problems, VRAS only-no domain login etc.) be logged with a notation of what the account is to be used for and with an expiry date and associated person for the maintenance of that account. These should be deleted once the task the id was requested for has been completed.

10.7. Service accounts are permitted, however, passwords should be changed at annual intervals and passwords should be made stronger than normal passwords . An application custodian or security manager should be responsible for managing these passwords. These passwords should not stored in cleartext or batch files or on a registry key and should not be written down.


B. Servers

1. Server Operating System Standards: All machines on the Praxair network (office or control side) must utilize an operating system that is supported by the provisioning vendor from a security patch and vulnerability perspective. Unsupported operating systems are not permissible unless the exception criteria below are completely met.

1.1 All servers operating systems need to meet the Praxair Server Operating System Standards

1.2 Standard: The following server operating systems are currently approved for use: Windows Server 2003 and Windows 2000 SP 4
1.3 Unless a waiver has been granted by Global IT Security, it is no longer permissible to run Windows NT 4.0 Server or any prior version of a Windows Server Operating System at Praxair

Microsoft Longhorn in beta or general availability form is not permissible for any reason at this time without the express, written consent of Global IT Security.

2. Server Configuration

2.1 All regions need to develop and employ their own server build checklist. All servers in that region should be built to the description in the checklist following good security practices and the other standards contained herein. Certain exceptions on configuration may be granted dependent upon the applications that are running on such systems. The burden of proof that the application will not perform as expected in these situations is always on the application owner and application anager. Fundamentally, all unnecessary services and processes on any server should be disabled or not loaded and the concept of least privilege applied.

2.2 Appropriate security templates need to be applied to each Windows server

2.3 Servers should only be configured and installed by IT server personnel or those whom IT management designates for configuration and installation on their behalf.

3. Server Hardening

3.1. Only the required services should be running on the servers. Services like telnet, ftp, www, sendmail need IT security approval before installation.

3.2. Service accounts should have the minimum privileges required to run the service. These accounts should be enumerated, managed and have their passwords known by as few people as possible. An application custodian or security manager should be responsible for managing these passwords. These passwords should not stored in cleartext or batch files or on a registry key and should not be written down.

3.3. Guest accounts need to be disabled and should not be used.

3.4. Any form of anonymous login should be steadfastly avoided

3.5. Banners should be set to not enumerate (provide back to the user in electronic form details of the versions of software on that server), where possible, the OS level or middleware being run on that server

3.6. Access to network shares, created specifically for IT use, should be given only to the specific users / groups requiring access. These shares should be reviewed annually by Global Information Security or, in some cases, an individual authorized by Global Information Security

3.7 Only machines approved by the Lotus Notes or the Global Information Security teams to act as SMTP gateways are permissible

4. Administrative Access

4.1. Administrative privileges need to be restricted to the administrators or for service/agent or network accounts, as appropriately identified by IT Infrastructure. These should be reviewed annually by the Global Director of Information Security.

4.2. Administrative functions should only be performed from a Praxair PC when connected to a Praxair network

4.3. When administrative privileges do not need to be utilized to accomplish tasks, the administrator should log into the network or device using their user account, not with their administrator account.

4.4. Passwords for administrative accounts should comply with Praxair Password Standards

4.5. Servers should never be left unattended while logged in with administrator access.


5. Departmental / Group Folders or Shares

5.1. A business owner should be identified for each departmental folder

5.2. The business owner specified is responsible for specifying access to folders; IT is accountable for the mechanism for maintaining access as specified

5.3. The business owner is responsible for alerting IT to any changes or revocations of access and IT is responsible for acting promptly on that information

5.4. Periodically, at worst annually, the business owner should review access lists as provided by IT for elimination of unneeded access and general clean-up

5.5. Use and creation of �public� drives is not permitted

C. Anti-Virus

1. Configuration

1.1. All Windows PCs and Servers must have IT Security approved virus scanning software installed and enabled at all times.

1.2. Praxair email servers should have IT security approved virus scanning software installed and enabled at all times. The e-mail servers should be configured to check for updates several times a day.

1.3. All virus scanners must be configured to log an alert to a central console when a virus is discovered.

1.4. All anti-virus software on PCs should be password protected (to the extent the feature is available) to prevent disabling / changing the configuration of software.

1.5. Users should not download or install an anti-virus program other than the one provided on their PCs without prior approval of IT Security

2. Administration

2.1 The status of anti-virus updates should be monitored on a daily basis on an operational level by the Infrastructure group and on a risk/threat evaluation level by IT Security

2.2 All PCs and servers must be routinely monitored to ensure the latest anti-virus pattern files have been successfully updated.

2.3 In case of high virus activity, the web sites of the anti-virus vendor should be monitored to ensure the latest pattern files are obtained and applied. This should be done a priority basis by IT Infrastructure and applied to critical servers first such as transaction processing and e-mail servers


D. Patch Management

1. Review and Analysis

1.1. Global IT security is responsible for reviewing all new security patches for applicability to the Praxair environment, assessing the severity and announcing the need to apply the patch to global administrators

2. Administration

2.1. Once notified, security patches must be applied by network administrators (for database patches, please see the database section of the standards) in a time-sensitive fashion based on the severity level. The severity level is to be determined by IT Security.

2.2. Security patches requiring testing should be carried out on a priority basis on devices that adequately mirror the production environment. If testing identifies issues with the patch, these issues should be brought to the attention of the IT Security group, who will coordinate with business and IT management on whether to proceed with the deployment of the patch

2.3. If patches require interaction by the system user such as a machine re-boot, this will be done as part of the normal patch procedure, but a pop-up message will appear indicating what the user should do or expect

2.4. Use of system tools for deploying patches is permitted, however the function of such tools should not be a gating factor in the deployment (i.e. if the tool is not working, the patches still need to be deployed)

2.5 Use of some scanning tools to determine missing or misapplied patches may be permissible. The tool that is most commonly recommended for this purpose is Microsoft Baseline Security Analyzer, which is available for free from Microsoft. Other tools are recommended for more in-depth scanning, but these can only be run by Global Information Security

E. Email and Instant Messaging


1. User Responsibilities- Email

1.1. When operating on your Praxair computer, employees must use only the company provided email system, Lotus Notes.

1.2 Notes ID files are critical components of email security and also to access the email system. Typically IT maintains backup copies of these files to ensure access in case of a hard disk crash of the user�s PC. However, if due to regulations or practice, IT has asked users to maintain backup copies of ID files, or if users want to do so for their own backup purposes, these files should be stored in a secure location with restricted access e.g. floppy diskette or CD-ROM in a locked cabinet.

1.3 Employees are required to mark emails �Business Confidential� where appropriate and utilize WinZip protected with a password when sending a confidential e-mail externally

1.4 Employees must not forward company email to personal accounts or use personal accounts for business purposes (e.g. Hotmail, Yahoo!)

1.5 Praxair Password guidelines for length and strength must be followed

1.6 Local replica�s of mail files must be encrypted

1.7 Use of utilities such as WinZip when sending large attachments to e-mail is encouraged


2. Instant Messaging- Standards and User Responsibilities


Instant Messaging is only authorized for communication within the Praxair network, including Praxair-issued Blackberry devices. The Praxair standard application for Instant Messaging (IM) is Lotus Notes Sametime.

While instant messaging chat features are also available from certain other, Corporate sanctioned applications, users are strongly encouraged to use only Sametime for Instant Messaging. Any use of chat features within those other authorized applications are subject to the Praxair Instant Messaging standards. The only other authorized applications are: - Groove, Netmeeting, and Meetingplace. Use of any other software for instant messaging (including freeware) like AOL, Yahoo, MSN, and Google Chat is not permitted.

F. Electronic Incident Management

1. All information security related incidents such as suspected theft of information, inappropriate content on computers or access to privileged information should be reported to IT Security with as much detail as can be provided. The reporter of incidents need not provide their personal information if they desire not to.

2. IT Security is obligated to perform an investigation of all reported incidents to determine the significance and impact of the exposure being reported.

3. IT Security participates in the incident tracking process and the forensic protection of evidentiary information related to computer or information exposures.

4. IT Security will involve other parties in such investigations as needed, including Corporate Security, other IT groups such as infrastructure or Lotus Notes, and Internal Audit.

5. Escalations of incidents that are unresolved and found to have a viable exposure are the responsibility of IT Security along with Corporate Security. Incidents are to be escalated in a timely fashion commensurate with the nature and evaluated impact of the exposure.

H. Database Standards

1. Configuration

1.1. Applications should be designed / configured to ensure users do not have access to the database by default. Only those with database administrative access should have full rights and authorization to make all changes and adaptations to the database. Database-system users other than the designated database administrator may be authorized by the application owner to access only a limited portion of the database, based on carefully evaluated needs. Other users may be allowed to issue queries, but not permitted to modify any data. It is the accountability of the application owner and the responsibility of the IT application manager to ensure that these authorization restrictions are not violated, but are reviewed.

1.2. Operating system patches on database servers should be given priority status by the nature of their underlying function on such servers and the importance of the data they are protecting

1.3. Database software patches should be evaluated by the regional/country DBA with a recommendation to deploy or not to deploy to the IT Manager. DB software patches should follow standard change control procedures.

1.4 The database administrator may authorize new users (based on input from the database owner), restructure the database and make changes necessary to maintain the on-going integrity of the database. These types of changes require an approved change control document. This type of access is analogous to that of a super-user for an operating system.

1.5 Security Guidelines for specified database technologies (e.g SQL, Oracle) should be adhered to

2. Administration

2.1. Administrative access to the databases should be restricted to the concerned database administrator and a backup

2.2. All database passwords are subject to the Praxair Password Policy

2.3. If users require access to the database for reporting access, such access should be authorized by the business owner of the application, should be read-only and be restricted only to the specified tables / rows/ fields the user requires access to.

2.4 All confidential information stored in external (non-Praxair managed) repositories must be kept in encrypted form within the third party data storage environment, with the configuration and layout of such environment approved by Global Information Security prior to it being used

L. Active Directory
    • 1.1 Nesting groups:
        • Nesting of groups is permissible, but not more than seven layers of nesting can occur.
        • Using nesting, you can add a group as a member of another group. You nest groups to consolidate member accounts and reduce replication traffic.
        • Nesting options depend on whether the domain functionality of your Windows Server 2003 domain is set to Windows 2000 native or Windows 2000 mixed.
        • Groups in domains set to the Windows 2000 native functional level or distribution groups in domains set to the Windows 2000 mixed functional level can have the following members:
        • Groups with universal scope can have the following members: accounts, computer accounts, other groups with universal scope, and groups with global scope from any domain. Groups with global scope can have the following members: accounts from the same domain and other groups with global scope from the same domain. Groups with domain local scope can have the following members: accounts, groups with universal scope, and groups with global scope, all from any domain. This group can also have as members other groups with domain local scope from within the same domain.

    • 1.2 Other Standards for Active Directory
        • Move your user and computer objects into an OU structure.
        • Keep the OU structure as simple as possible.
        • Do not nest OUs more than 10 layers deep.
        • Numbers of OUs must be kept to a minimum.
        • Group Policy to groups through Group Policy Filtering.
        • Use domain local groups to control access to resources, and use global groups to organize similar groups of users.
        • Don't simply delete and re-create groups on the fly because each group SID is unique.
        • Don't include users from other Mixed mode domains in a forest in universal groups.
        • Don't utilize local groups for permissions in a domain environment.

Must allow for these data elements to be stored, backed-up and transmitted in encrypted mode using an industry acceptable algorithm, such as AES, 3DES, SSL or Blowfish.
